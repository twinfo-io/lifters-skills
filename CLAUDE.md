# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

`lifters-skills` is the central repository of Claude Code skills for Lifters teams. It serves two purposes:

1. **Skills distribution** — Claude Code commands (`/lf-discovery`, `/lf-briefing-ux`, `/lf-new-feature`, `/lf-design-system`) installed company-wide via `npx skills add twinfo-io/lifters-skills`, available in every project on the developer's machine.

2. **Specification workspace** — Houses the `ai/specs/` directory where feature specs for this repository itself are maintained.

## Format

This repository follows the **Agent Skills** standard (`agentskills.io`). Each skill lives in its own directory under `skills/` with a `SKILL.md` file containing YAML frontmatter and the skill instructions.

## Installation

```bash
npx skills add twinfo-io/lifters-skills       # via registry (recommended)
curl -sSL https://raw.githubusercontent.com/twinfo-io/lifters-skills/main/install.sh | bash  # fallback
```

Skills are installed to `~/.claude/skills/` (user-level) — not project-level.

## Commands

### `/lf-discovery [feature description]`

Starts an interactive feature discovery. Collects input documents first (URLs, pasted content, multiple docs), then creates the folder structure, conducts a greenfield/brownfield interview, researches market references automatically, and generates `discovery.md`.

```
/lf-discovery "Cobrança recorrente com Stripe"
/lf-discovery   # without argument — asks for description first
```

Output: `ai/specs/YYYYMMDDHHmmSS_nome/discovery.md`

### `/lf-briefing-ux`

Reads an existing `discovery.md` and generates the UX/UI Briefing — the artifact for the design team to start prototyping without needing a full technical briefing. Covers personas, screen navigation map, per-screen specs with states and ASCII wireframes, user flows, microcopy, and visual references. Requires `discovery.md` to exist.

```
/lf-briefing-ux
```

Output: `ai/specs/YYYYMMDDHHmmSS_nome/briefings/briefing-ux.v0.md`

### `/lf-new-feature`

Reads an existing `discovery.md` (or conducts inline discovery if none exists) and generates the three canonical technical artifacts. If `briefing-ux.vN.md` exists, uses it to populate the personas and UX sections of the technical briefing without repeating questions.

Output:
- `ai/specs/YYYYMMDDHHmmSS_nome/briefings/briefing-tech.v0.md`
- `ai/specs/YYYYMMDDHHmmSS_nome/specs.md`
- `ai/specs/YYYYMMDDHHmmSS_nome/wps.md`

### `/lf-design-system ["Design System Name" figma-url]`

Connects to Figma via MCP Server, extracts all design tokens (typography, colors, spacing, border radius, shadows) and generates `specs/design-system.md` — the project's official visual source of truth. Requires the Figma MCP Server to be configured and authenticated.

```
/lf-design-system "Atlas" https://figma.com/design/abc123/Atlas?node-id=3-281
/lf-design-system   # without arguments — asks for name and URL
```

Output: `specs/design-system.md`

## Cross-reference Headers

All generated briefings include headers that reference their upstream artifacts:

- `briefing-ux.vN.md` references `../discovery.md`
- `briefing-tech.vN.md` references `../briefings/briefing-ux.vN.md` + `../discovery.md`

On each refinement (v1, v2...), headers are updated to reference the most recent files available at generation time, creating a complete audit trail.

## Canonical Format Reference

All generated files must follow the format of these reference files — **do not modify them**:

- `ai/specs/20260323142630_google_docs/briefings/briefing-ux.v0.md` — UX/UI briefing quality reference
- `ai/specs/20260323142630_google_docs/briefings/briefing.v0.md` — technical briefing quality reference
- `ai/specs/20260323142630_google_docs/specs.md` — spec format reference
- `ai/specs/20260323142630_google_docs/wps.md` — work package format reference

**specs.md** — 12 sections per SPEC-XX:
Objetivo · Contexto · Personas e Papéis · Comportamento esperado · Regras de negócio · Contrato de API · SLA e Performance · Observabilidade · Critérios de aceite (DADO/QUANDO/ENTÃO) · Estado atual · Mudanças necessárias · Definição de pronto

ID · Spec relacionada · Tipo · Estimativa · Dependências · Pode paralelizar com · Testes requeridos · Status

File footer (mandatory): Mapa de Dependências · Riscos e Pontos Desconhecidos · Oportunidades de Paralelização

## Adding New Skills

1. Create `skills/skill-name/SKILL.md` with YAML frontmatter (`name` + `description`) and the skill instructions
2. Add templates to `skills/skill-name/templates/` if needed
3. Add the entry to `skills.json`
4. Document in `README.md` and `CHANGELOG.md`
5. Open a PR for team review

**Required frontmatter:**
```yaml
---
name: skill-name
description: What it does and when to use it (max 1024 chars, no XML tags)
---
```

Naming convention: `kebab-case`, imperative English verbs (`gen-docs`, `review-pr`).
